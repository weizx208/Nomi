# 多维体验质感审计（Experience-Feel Audit, E5）

> 2026-06-20 ｜ 状态：**已拍板，执行中**
> **用户决策**：A=建探针基建（要真实 ms/FPS 数字）｜B=只测 UI 反馈质量（不花额度跑真实生成）｜C=全 J1–J5 覆盖
> 起因：用户希望在「功能能不能跑通」之外，做一次更高维的测试——**等待时间、体验丝滑、视觉美感、舒不舒服、有没有不爽**。本文把这几个直觉维度对应到成熟方法论，定义可量化信号、AI 驱动采集方式、评分与执行编排。

---

## 0. 这到底是什么测试（给它一个名字）

用户描述的「更多维的测试」在业界有成熟对应，它**不是功能 pass/fail 测试**，而是**专家启发式 + 感知性能的体验质感审计**：

- **启发式评估（Heuristic Evaluation, Nielsen 1994）**：专家拿一套公认原则逐屏挑毛病，按严重度 0–4 打分。3 个评审能抓到约 60% 的可用性问题，成本远低于真实用户测试。
- **感知性能（Perceived Performance）**：关注「感觉有多快」而非「真有多快」——Doherty 阈值 400ms、RAIL 模型、INP≤200ms、骨架屏/乐观更新。
- **情感/欲求度（Microsoft Reaction Cards, 2002）**：用情感词捕捉「爽不爽、舒不舒服、想不想用」这种纯启发式量不到的层面。

> 这正是 R13「穿透式体验走查」的**升级版**：R13 现在判断「顺不顺、美不美」是定性人眼判断；本方案把它**结构化 + 可量化 + 可回归**，让「丝滑」「等待」这种模糊感受有数字背书。
> 业界已有用 LLM 在开发阶段自动识别可用性缺陷的实践（arXiv 2512.04262），即「AI 当评审员」——我们的常驻驱动 + 截图感知正好是这套的执行体。

**命名**：本项目内称 **E5 体验质感审计**（E5 = 用户提的 5 个维度）。

## 1. 五维度 → 方法论 → 可量化信号映射

| # | 用户的说法 | 正式维度 | 方法论锚点 | AI+埋点能采的客观信号 | 主观判断 |
|---|---|---|---|---|---|
| **D1** | 等待时间 | 感知性能 | Doherty 400ms / RAIL Response / INP≤200ms | ① click→首视觉反馈(ms) ② click→内容就绪(ms) ③ 有无即时反馈(按下态/skeleton/乐观更新) ④ 长操作有无进度 | 等待时焦虑吗？知道在等什么吗？ |
| **D2** | 体验丝滑 | 响应性 & 运动质量 | RAIL Animation 16ms / 60fps / Long Animation Frames API | ① 动画/拖拽/滚动期间 FPS ② long task(>50ms) 次数 ③ 掉帧峰值 ④ 缓动曲线是否自然 | 拖起来跟手吗？动画突兀吗？ |
| **D3** | 视觉美感 | 视觉设计审计 | 视觉层级 / Gestalt / 对齐 / 留白节奏 / 对比度 WCAG / token 一致性 | ① 关键元素对齐实测(getBoundingClientRect) ② 文字对比度比值 ③ token 合规(check:tokens) ④ 间距是否走 token 节奏 | 截图专家评审：层级清楚吗？高级吗？ |
| **D4** | 舒不舒服 | 认知负荷 & 人体工学 | Hick / Miller / Tesler 定律 / 识别优于回忆 / 渐进展开 | ① 每屏可点元素数 ② 单次决策选项数 ③ 首屏信息密度 ④ 默认值是否预填 | 一眼知道干嘛吗？要不要动脑？ |
| **D5** | 有没有不爽 | 启发式 & 摩擦/愉悦 | Nielsen 10 启发式 + 严重度 0–4 / Microsoft 反应卡 | ① 摩擦点(死胡同/无反馈/意外行为)计数 ② 错误可恢复性 ③ 反应卡情感词命中 | 哪一步想骂人？哪一步惊喜？ |

## 2. 评分体系（让模糊感受有数字）

**采分单元** = `旅程步骤 × 维度`。每个单元产出：

```
{ journey: "J1", step: "拆镜头→落画布", dimension: "D1-等待",
  measured: { firstFeedbackMs: 120, contentReadyMs: 14200, hasSkeleton: false, hasProgress: true },
  heuristic: "Doherty: 首反馈<400ms ✅；长操作>1s 有进度 ✅",
  severity: 1,               // Nielsen 0=非问题 1=外观 2=小 3=大 4=灾难
  evidence: "shots/j1-split-wait.png",
  verdict: "首反馈快；但 14s 生成期进度文案是静态'…'，不知道剩多久" }
```

**汇总产物**：
- **严重度 × 维度 热力图**（哪个维度最痛一眼看出）
- **Top-N 不爽清单**（severity≥3 优先）
- **感知性能数字表**（每个关键动作的 ms 实测，对照 400ms/1s 阈值）
- **反应卡情感画像**（整体调性：高级/克制/廉价/混乱…）

## 3. AI 怎么真采到这些信号（关键：不靠拍脑袋）

现有常驻驱动（`tests/ux/ui.mjs`）只能 snap/shot/click。要量化 D1/D2 需给驱动**加三个埋点探针**（一次性基建，可复用回归）：

| 探针 | 采什么 | 实现 | 落点 |
|---|---|---|---|
| **延迟探针** | click→下一次有意义重绘的 ms | 包裹动作：发起前 `performance.now()`，用 `requestAnimationFrame` + MutationObserver 抓首次 DOM/像素变化 | 新增 `ui.mjs timed-click` |
| **帧率探针** | 一段时间窗的 FPS / long task | 注入 `PerformanceObserver({type:'longtask'})` + `long-animation-frame` + rAF 计帧；动作前 start、后 stop 读数 | 新增 `ui.mjs fps-start/fps-stop` |
| **几何/对比度探针** | 对齐偏差、对比度比值、可点元素数 | 纯 eval：`getBoundingClientRect` 批量对齐检测 + `getComputedStyle` 算前景/背景对比度 | 扩 `snap` 输出 |

> D3 美感 / D5 情感仍是**截图 + 评审 agent 人眼判断**（这部分本就该主观），但 D1/D2/D4 用真实数字背书，避免「我觉得有点卡」这种说不清的结论。

## 4. 执行编排（阶段 × 产出）

| 阶段 | 做什么 | 产出 | 是否需写码 |
|---|---|---|---|
| **S0 基建** | 给常驻驱动加 3 个探针（延迟/帧率/几何对比度） | `tests/ux/ui.mjs` 扩命令 + 一个 `probe.mjs` | ✅ 需写码（≤1 文件，约 150 行） |
| **S1 走查采集** | 清场→build→起驱动→逐旅程跑，每步采 5 维信号+截图 | 原始数据 JSON + 截图集 | 否（用 S0 工具） |
| **S2 专家评审** | 多视角 agent 对截图做 D3/D5 评审（设计师/真实用户/情感反应卡） | 评审结论合并 | 否（agent） |
| **S3 汇总打分** | 合并客观信号+主观评审→严重度热力图+Top-N | `docs/audit/2026-06-20-experience-feel.md` | 否 |
| **S4 修复拍板** | P0/P1 当场修；方案级取舍出对比表给用户 | 修复 + 回归断言 | 视发现而定 |

**范围缩放**：先跑**最高频的 J1（主链路）+ J3（冷启动）**两条，验证方法有效、产出有料，再决定要不要补 J2/J4/J5。

## 5. 需要你拍板的岔路（R3）

### 决策 A：要不要先建「探针基建」(S0)？

| 方案 | 你得到什么 | 代价 |
|---|---|---|
| **A1 建探针（推荐）** | 等待/丝滑有真实 ms+FPS 数字，能锁回归断言，下次重构能复跑 | 先写 ~150 行驱动扩展（半天） |
| A2 不建，纯人眼 | 快，直接走查截图判断 | 「卡不卡/等多久」只能定性描述，无数字、不可回归——容易重蹈「我觉得有点卡」 |

### 决策 B：生成等待（D1 大头）怎么测？

真实生成要花你的额度且 10–60s 不等。三选一：

| 方案 | 测到什么 | 代价 |
|---|---|---|
| **B1 只测 UI 反馈质量（推荐）** | 生成期的 skeleton/进度/可否继续操作/能否取消——感知性能的核心 | 不花额度；测不到真实 vendor 时延 |
| B2 真跑 1–2 次生成 | 含真实端到端时延的完整体感 | 花额度（你的资源） |
| B3 两者结合 | 最全 | 花额度 |

### 决策 C：先跑几条旅程？

| 方案 | 覆盖 | 代价 |
|---|---|---|
| **C1 先 J1+J3（推荐）** | 主链路+冷启动，最高频 | 范围小、出结论快，验证方法后再扩 |
| C2 全 J1–J5 | 全覆盖 | 一轮很长 |

## 6. 不动什么 / 回滚

- **不改任何产品代码**（S0 只动 `tests/ux/` 测试工具，不进 src/electron 产品路径）
- 探针纯注入式（eval/PerformanceObserver），不改 app 行为
- 回滚 = 删 `tests/ux/probe.mjs` + 还原 `ui.mjs`，零产品影响

## 7. 验收门

- 每个维度至少 1 条带**证据（截图/数字）**的结论，不许空泛
- D1/D2 结论必须附实测 ms/FPS，对照阈值（400ms / 60fps）
- 严重度热力图 + Top-N 不爽清单成型
- 落 `docs/audit/2026-06-20-experience-feel.md`
