# Seedance 2.0 接入更新：4K 清晰度 + Mini 变体（kie.ai / apimart）

> 2026-06-24。用户：「Seedance 2.0 上了 4K，apimart 和 kie 都得更新；好像还上了 Seedance 2.0 Mini，看它们有没有，有就整进来。」

## 背景 / 真实摩擦

ByteDance 在 2026 火山引擎 FORCE 大会给现役 **Seedance 2.0 系列**追加了**原生 4K** 输出（不是 2.5、不是 4.0——4K 是 2.0 的就地升级），并放出**轻量 Mini 档**（比 Fast 更快更便宜，定位高频草稿/迭代）。Nomi 通过两个中转接 Seedance：**kie.ai** 与 **apimart**。现接入的清晰度上限钉死在 1080p，Mini 还没有——用户付费买了 4K 却在 Nomi 里选不到，等于能力黑洞。

## 已核实的供应商真相（照官方 API 文档全覆盖，非凭记忆）

逐条对着 live 文档核过（R5），两边**确实不一样**，必须分别处理：

| 维度 | kie.ai (`bytedance/seedance-2`) | apimart (`doubao-seedance-2.0`) |
|---|---|---|
| 文档 | docs.kie.ai/market/bytedance/seedance-2 | docs.apimart.ai/.../doubao-seedance-2-0/generation |
| 4K | `resolution` 枚举含 `4k`（480p/720p/1080p/**4k**），标准档支持 | `resolution` 含 `4k`，**仅基础档 `doubao-seedance-2.0` 支持** |
| 1080p | 标准档支持 | 仅 `doubao-seedance-2.0` 与 `-face` 支持 |
| Fast 档 | `bytedance/seedance-2-fast`：仅 480p/720p | `-fast` / `-fast-face`：仅 480p/720p |
| **Mini** | **有**：`bytedance/seedance-2-mini`，仅 480p/720p，能力同档 | **无**——apimart 自家文档不存在任何 mini 变体 |

**两个文档坑（已核实）**：
1. kie 的 mini 文档页（docs.kie.ai/market/bytedance/seedance-2-mini）枚举值疑似从 fast 页克隆——强制 enum 写成 `bytedance/seedance-2-fast`，但描述文字明确写「Must be `bytedance/seedance-2-mini`」。**以描述声明的 `bytedance/seedance-2-mini` 为准，并以 live API 实跑核实**（落地前必须一次真实 createTask 验证该 model id 被接受，否则回退）。
2. apimart「4k」在部分二手资料里被混说成 Seedance 4.0 的特性——已对 apimart **自家** generation 文档核实：`doubao-seedance-2.0` 基础档 `resolution` 确实含 `4k`（「Ultra HD (only supported by doubao-seedance-2.0)」）。

## 范围

- **kie.ai**（`src/config/modelArchetypes/seedance.ts`）：
  - 标准档 `resolution` 加 `4k` → `[480p,720p,1080p,4k]`。
  - 新增 **mini 变体**（`bytedance/seedance-2-mini`），清晰度收窄 480p/720p（与 fast 同收窄逻辑，复用一份 override）。
  - archetype + variant 的 `identifierPatterns` 收纳 mini 串（旧/显式 modelKey 可归一）。
- **apimart**（`src/config/modelArchetypes/seedanceApimart.ts`）：
  - 基础档 `resolution` 加 `4k`（`PARAMS` = 全能力 `[480p,720p,1080p,4k]`，沿用「PARAMS=max，变体往下收窄」既有范式）。
  - `face` 变体新增收窄 = `[480p,720p,1080p]`（保 1080p、去 4k，符合 apimart 约束）。
  - `fast`/`fast-face` 维持 `[480p,720p]`（不变）。
  - **不加 Mini**（apimart 没有；诚实标注，不造假变体）。
- 测试：`index.test.ts` 清晰度/变体断言更新 + mini 解析；kie e2e 加 mini 落地 + 一次 live 核实。

## 不动项（非目标）

- 不碰 duration（文档 4–15 vs 表 5–15 的轻微不一致是既有问题，不在本轮）。
- 不接 Seedance **2.5**（企业 beta，公测约 7 月，规格未官方定型）、不接 **4.0**。
- 不改传输层 catalog body（`model` 已是 `{{request.params.model}}`，变体自动带 modelKey，零传输改动）。
- 不新增 catalog 模型行（Mini 是 variant，骑现有 `seedance-2` 行；同 fast）。

## 设计要点

- **P1 无并行版**：4K 只是给现有 select 加一个选项；Mini 只是 `variants[]` 多一项 + 复用既有 `narrowResolutionToFast` 这条收窄原语（fast 与 mini 共用，不复制）。kie mini 若 live 核实失败（只接受 `-fast`）→ **不落 mini 变体**（否则就是 fast 的并行重复版），改为如实回报。
- **P4 通用第一**：4K/Mini 都走通用变体轴与参数控件，不为它们写专属 UI。
- **per-variant 约束建模**：apimart 4k 是基础档独占，故 `face` 必须显式去 4k——用与 fast 同构的 override 函数表达，保证 UI 选项收窄与发送一致（`specializeArchetypeForVariant` 纯函数，渲染与构造同源）。

## 回滚

纯档案/测试改动，无迁移、无落盘 schema 变化。`git revert` 即可；旧项目节点 modelKey 不受影响（patterns 只增不减）。

## 验收门

1. 五门全过（filesize→tokens→lint→typecheck→test→build）。
2. 单测：kie 标准含 4k、fast/mini=480/720、mini 解析到 `seedance-2`；apimart 标准含 4k、face=[480,720,1080] 无 4k、fast=[480,720]。
3. **live 核实**（额度默认授权）：apimart 基础档 `4k` 真 createTask 被接受；kie `bytedance/seedance-2-mini` 真 createTask 被接受（不接受则回退、不落 mini）。
4. R13 真机走查：画布选 Seedance（两中转）→ 清晰度下拉出现 4k；kie 变体下拉出现 Mini；截图人眼确认。
