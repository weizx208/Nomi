# Nomi 项目工程纪律 — 给 Claude 的读取项

> 每个会话开始**强制读这份**。先读「总纲」与「操作手册」（路由 + 决策自治 + 自检 + 教训），再按需查 14 条规则——它们都是总纲的落地。违反任一条 = 工作错误。

## 总纲（北极星）：长期价值优先，别停在短期"先解决"

> 每个决策先问一句：**这是为长期做对的事（架构、技术栈、模块化、可维护性），还是为眼前省事打的补丁？**
> 短期补丁堆起来就是下一轮的债——`styles.css` 8700 行死代码、双目录漂移，都是当年"先跑起来再说"留下的账，后来花了整段会话才还清。
>
> 下面 14 条规则全是这条总纲的具体落地，按性质分三类（**不改编号，只是看清彼此关系**）：
>
> **一、想清楚再动手**（调研 → 评审 → 模块化，这一类就是"长期架构"本身）：
> 规则 **5**（查库官方文档）→ **6**（读顶尖开源架构）→ **9**（想模块化分层）→ **7**（六角色评审）→ **8**（用户可见的先出样张）→ **3**（取舍出对比表给用户拍板）。
> 其中 **9 是 7 在"架构决策"上的应用**，**8 的评审步就是 7**——它们不是新东西，是同一件事的不同切面。
>
> **二、动手时的纪律**（写码不留债）：
> 规则 **1**（加新必删旧）——**10 是它在样式上的实例**；规则 **4**（多文件先写文档）。
>
> **三、UX 标准**：规则 **2**（用户视角 + 极简）。
>
> **拿不准时回到这条：选长期对的，不选短期快的。** 该不该停下来调研/评审？→ 停。冒出"先打个补丁以后再说"？→ 那个"以后"就是欠债。

## 操作手册（总纲之后、规则之前先看；目的：自动按规则走 + 少打扰用户）

> 这是「快速路由层」：让 Claude 自己判断**什么时候用哪条规则、什么自己决、什么才找用户拍板**，
> 不必每次等用户提醒。规则细节仍看下面 14 条；这里只给触发与决策。**遇到下面没覆盖的新情境，
> 自己按总纲推断 + 顺手把该情境补进本手册**（这一节随用随长，是活的）。

### A. 决策自治：默认自己定，只把"最重要、最小"的拿给用户拍
- **自己定**（做完一句话说明理由，用户可回退）：实现细节、类型/组件复用、模块怎么拆、命名/文案、
  测试怎么写、bug 修复顺序、哪条规则适用、样张的非方向性细节。
- **才找用户拍**（用 AskUserQuestion，**合并成一轮**、给推荐项；范围大的配对比表）：
  1. 真·产品方向 / 取舍，用户视角差异大或**不可逆**（词汇策略、是否花额度真实生成、破坏性操作）。
  2. 架构岔路：影响大、回滚贵、有多个合理解 → 对比表（规则 3）+ 推荐项。
  3. 需要用户独有之物：API key、额度、外部账号、真实素材。
- **别一次问一堆**：能推断的先定，只留真正要拍的几条；问题尽量并成一轮，每条给推荐项。

### B. 规则触发路由（情境 → 自动适用哪条）
| 情境 | 自动适用 |
|---|---|
| 加新组件/类型/流程 | 先找现成可复用、**别造第二份真相源/并行版**(1,9)；新替旧→同 commit 删旧(1)；样式删旧 CSS(10) |
| 碰第三方库/框架 | 先查官方文档(5) + 读顶尖开源真实代码(6)，不手搓 |
| 用户可见的 UI/文案 | 先出可视样张(8) + 设计/真实用户 Agent 审(7)；用户视角 + 极简(2) |
| 架构/分层/状态归属 | 先想模块化(9) + 拉 CTO/前端/后端 Agent 讨论(9/7)；有取舍出对比表(3) |
| 项目级方案定稿 | 6 角色评审(7) |
| 多文件/多步改动 | 动手前写 docs/plan 或 docs/audit 文档(4) |
| 写/改任意 .ts/.tsx | ≤800 行；巨壳白名单只减不增(12) |
| 改动完成 + 验证过 | build+typecheck+test+filesize 绿 → 自己 commit + push(11) |
| 声称"用户可见改动验证过" | Playwright `_electron` 穿透走查，不只单测(13) |
| 距上次审计 ≥25 commit / 发版前 | 多维 subagent 审计(14) |

### C. 动手前自检（自动跑，不等提醒）
1. 有现成的能复用吗？我是不是在造**第二份真相源 / 并行版**？(1,9)
2. 加了新的 → 旧的删了吗？(1,10,12)
3. 碰库了 → 查官方文档 / 开源了吗？(5,6)
4. 用户可见 → 出样张了吗？**渐进展开**（收下拉/弹层，别平铺一长条）、**防遮挡**（控制项别盖住它要改的对象）、**空间极简**（别留大空框）？(2,8)
5. **通用第一**：这东西对所有人/所有供应商通用，还是只服务某特例？
6. 文件要长大 / 喂巨壳吗？(12)
7. 改完绿了吗？该自己 commit + push 吗？(11,13)

### D. 本会话沉淀的教训（已成默认，违反即工作错误）
- **通用第一**：能力/模板按「模型身份」认，**与供应商解耦**——任何人经任意供应商接入同一模型都用同一套；
  认不出走**通用回退**（按接入文档原样展示、不替用户合并、不藏能力）。绝不因为我们做了合并就挡住自由接入。
- **复用优先**：新参数/控件先看 `modelCatalogMeta` 等现有类型再决定，别另造一套（规则 1/9 的高频踩点）。
- **UI 收纳**：参数收进下拉/弹层渐进展开；切换器与它控制的输入要**同屏可见**（防遮挡）；输出节点与参数框分离；空框收成占位。
- **词汇统一**：跨模型用统一意图词为主、vendor 原词为副标签。

## 规则 1：加新必删旧（No Parallel Versions）

**当我引入一个新组件 / 新流程替代旧的：**

- 同一 commit 里**删除被替代的旧代码**
- 不留"逃生口 / Collapse / 备选 fallback / 高级开关"——这些都是借口
- 死代码（grep 不到外部引用）立刻物理删除文件，不留"以防万一"
- 反例：M5.3 我加了 `OnboardingWizard` 但留了 `ModelCatalogImportSection`，结果用户看到两套 UI 混杂。错。
- 正例：今天的 `chore: codebase cleanup (audit 2026-05-30 section A + C)` —— 移动文件同步更新所有 import，0 残余。

**如果"不敢删"：**
- 真的不确信新方案 → 不要 ship 新方案
- 怕回归 → 加测试，不是留旧代码
- 旧的有点价值 → 把那点价值合并到新代码里，再删旧

## 规则 2：用户视角 + 极简

**做 UI / 文案 / 卡片设计前，先按这个顺序问：**

1. **用户进来要看什么？** 列出每条信息，按重要度 ⭐⭐⭐ / ⭐⭐ / ⭐ 排序。
2. **每条信息有用户行动价值吗？** 没有行动价值的信息 = 噪音 = 删。
3. **0 权重的常见嫌疑犯：**
   - 节点功能描述文字（"承接上游提示词生成图片。"）
   - onboarding 文案在每次出现（"或选中后输入提示词生成"）
   - 重复的分组标签（节点已在 "角色" group 里，节点头还写"角色"）
   - 长 error stack 灌满卡片（应缩成 ⚠️ + tooltip）
   - 双层 border / 圆角套圆角（违反 Design.md 第 6 节）
   - 永远 80px 高的"信息区"即使内容为空
4. **设计系统**：参考 `Design.md` + `src/design/`，token-only，光模式，密度优先。

**反例**：M5.3 wizard 首版我写"高级 / 专家逃生口"折叠区，留着 600 行旧代码。错。
**正例**：今天角色卡删了"角色"tag、内层 border、"或选中..."文案、用户不需要的描述行。

## 规则 3：决策格式

涉及范围 / 取舍时，**先给用户用户视角的对比表**，再让用户拍板：

```
| 方案 | 用户看到什么 | 代价 |
|---|---|---|
| A | ... | ... |
| B | ... | ... |
```

不要单方面"我建议 A"然后开干。

## 规则 4：执行前必写文档

涉及多文件 / 多步骤的改动，先在 `docs/plan/` 或 `docs/audit/` 写一份执行文档，含：
- 范围
- 不动什么
- 回滚策略
- 验收门

写完用户能预读 / 反驳；执行完回填结果。

## 规则 5：碰框架 / 库，先查官方文档再写（Context7 强制）

> 反复出现的硬伤：AI 自己手搓一套，而不是用库的官方实现。例：手写 viewport
> transform，而 React Flow 早就有 `screenToFlowPosition` / `zoomOnScroll` /
> 受控 `viewport`。这种"看起来能跑、其实跟生态脱节"的代码 = bug 温床 + 维护债。

**铁律**：凡是涉及第三方框架 / 库的实现或改动（React Flow、AI SDK、Mantine、
Electron、Tiptap、Vite…），**动手写之前必须先用 `context7` 查该库的官方文档 / 推荐
用法**，再按官方实现来写。已配置 Context7 MCP（项目级 `.mcp.json`）。

> **Context7 接入坑（2026-06-03 定位）**：项目级 `.mcp.json` 的 MCP 服务器**必须经用户批准**
> 才会把工具加载进会话——`claude mcp list` 显示 `✓ Connected` 只是可达性探测，不代表工具可用。
> 根因是 `~/.claude.json` 里该项目的 `enabledMcpjsonServers` 没列 `context7`。已加进去（下个会话生效）。
> 若 `ToolSearch "resolve-library-id"` 仍找不到工具 → 在 Claude Code 里跑 `/mcp` 批准，或退回 WebFetch 查 `reactflow.dev` / `tiptap.dev` 等官方站（等价）。**不是"下个会话自动生效"，是审批问题。**

流程：
1. `resolve-library-id` 拿到库的 Context7 ID（如 React Flow → `xyflow/xyflow`）。
2. `get-library-docs` 拉该库相关主题的官方文档（带上 topic，如 "viewport zoom"）。
3. 对照官方推荐的 API / 模式实现；**不要凭记忆或臆测手搓**。
4. 若官方就是没有现成能力，才允许自定义，并在注释里写明"官方无此能力，故自实现"。

**不查就写 = 工作错误**，等同违反规则 1（手搓的等于引入了一套与官方并行的劣质实现）。

## 规则 6：先研究顶尖开源项目的真实代码，再做方案（不重复造轮子）

> 规则 5 用 Context7 查"库的官方 API"；这条更上一层——查"**架构与思路**"。
> 反复出现的硬伤：我凭记忆/印象就出方案，而没去看业界顶尖项目**真实代码**怎么解的。

**铁律**：做任何项目方案前，先去 GitHub 找 1~N 个**顶尖开源项目**，读它们的**真实代码/实现**（不是只扫 README、不是凭印象），产出：
- 它们**怎么做的**（具体到模块/数据结构/流程，给出文件/代码位置）
- 我们**能直接借鉴什么**
- **哪里不适用**、为什么

能借鉴成熟实现就别重复造轮子。允许的"自创"只在确认无现成方案后，且写明理由。
本地 agent runtime 方向的参考池（示例，非穷尽）：Cline、OpenHands、Aider、Continue、
opencode、Cherry Studio、Jan、LobeChat；画布/节点向：tldraw、ComfyUI、xyflow。

## 规则 7：项目方案定稿前，必过 6 角色子 agent 审查

**触发条件**：任何涉及架构 / 取舍 / UX 的"项目方案"，在给用户拍板前。

**铁律**：用 Agent 工具创建子 agent，从以下 **6 个角色各审一遍**，再汇总：
1. **CTO** —— 架构合理性、可维护、技术债、扩展性、与现有内核是否冲突
2. **设计师** —— 视觉/一致性、是否对齐 `Design.md` 与 `src/design/`、密度优先
3. **产品经理** —— 用户价值、优先级、范围是否过大、是否解决真痛点
4. **前端** —— 实现可行性、状态管理、性能、组件复用
5. **后端** —— runtime / 数据 / 安全 / IPC / 持久化影响
6. **真实用户**（最终综合体验）—— **不只是代码，而是设计 + 体验 + 界面的综合体感**：
   "我作为用户，用起来顺不顺、爽不爽、看着舒不舒服"

**流程**：研究开源（规则 6）→ 起草方案 → 6 角色并行审查 → 汇总意见、修方案 →
带对比表（规则 3）给用户拍板。目的不是只验"代码对不对"，而是**综合体感过关**。

## 规则 8：用户可见的设计改动，先出"可视样张"再做

> 反复教训：直接动手做 UI，做完才发现方向/样子不对，返工。光文字描述方案，用户也想象不出最终长啥样。

**触发条件**：任何**用户会看到**的东西——界面 / 卡片 / 节点 / 面板 / 交互 / 文案排版。

**铁律（动手实现前必须走完）**：
1. **参考设计系统**：`Design.md` + `docs/design/nomi-design-system.md` + `src/design/`（token、现有组件），不凭空设计、不手搓重复组件。
2. **出"可视样张"——最低成本让用户先看到样子**：优先一张 **HTML mockup**（或最简化的真实实现 demo），而不是只有文字。用户能直接打开看到长相再决定。
3. **评审**：设计师 Agent + 真实用户 Agent（规则 7）过一遍。
4. **用户本人确认**后，才进入实现。

**不出样张就开干 = 工作错误。**

**复杂交互（如时间轴 图片轨/媒体轨）**：尤其不能"一次性出发就干"。要**分多轮**：技术选型/框架调研（用 Context7 查核心框架官方做法，规则 5/6）→ 梳理功能列表 → 样张 → 评审 → 补充调整 → 再定方案 → 才实现。持续迭代，别图快。

## 规则 9：写码前先想模块化 / 架构（别把东西堆在一处）

> 反复出现的硬伤：上来就往一个文件 / 一个组件 / 一个 store 里塞，越滚越大，最后所有东西
> 混在一处——改一处怕动全身，死代码没人敢删，关注点全纠缠。

**铁律**：任何涉及新模块 / 新组件 / 新流程，或往现有大文件继续加东西之前，**先停下来想清楚整体的模块化与架构**：
- 这块逻辑**该放哪一层**？（UI / 状态 / 领域逻辑 / runtime / 持久化——别让它们混在一个文件里）
- 关注点是否分离？渲染归渲染、状态归 store、领域逻辑归领域层、副作用归 runtime。
- 是不是又在往一个巨壳里塞？（反例：`BaseGenerationNode.tsx` 1354 行什么都管；`styles.css` 8867 行混了多代画布。**这就是没先想架构的代价**。）
- 新东西和现有内核的边界在哪？会不会引入第二份真相源 / 并行实现（连回规则 1）。

**架构决策必须过讨论 + 调研，不能一个人拍**：
1. **拉角色 agent 一起讨论**——模块化 / 架构问题至少过 **CTO + 前端 + 后端**三角（用 Agent 工具角色扮演，规则 7 的子集）：分层是否合理、状态放哪、IPC / 持久化 / 数据边界是否清晰、可维护可扩展。
2. **做调研、查官方文档**——涉及框架 / 技术栈选型，先用 Context7 查官方推荐架构（规则 5），再读顶尖开源真实代码怎么分层（规则 6），**确保整个框架 / 技术栈的用法没问题**，不是凭记忆手搓一套。
3. 有取舍时出对比表给用户拍板（规则 3）。

**不先想模块化就开写 = 工作错误**，等同违反规则 1——堆出来的巨壳和混在一起的代码，就是下一轮要还的债。

## 规则 10：样式一律 Tailwind 写组件上，CSS 文件只可减不可增

> 反复出现的硬伤：组件迁去 Tailwind 了，旧 CSS 却没删——堆成 `styles.css` 8867 行死代码，
> 分 Tier 1/2/3 才清完（约 8700 行全是没人消费的尸体）。这是规则 1「加新必删旧」最贵的反例。

**铁律（碰样式时）**：
- **新样式一律用 Tailwind utility 写在组件 `className` 上**（utility-first，对齐 Tailwind 官方）。
  不在全局 CSS 里新建组件类。
- **不用 `@apply` 把 utility 重新打包成类**（官方不推荐）；重复就**抽组件 / template partial**，
  样式和结构就近放一起。
- **CSS 文件分工固定，且只可减不可增**：
  - 设计 token → `src/theme/nomi-tokens.css`（`--nomi-*`）/ Tailwind `@theme`。
  - 全局 reset / `body`/`#root` / keyframes → `src/styles/index.css` 或 `tailwind.config.ts`。
  - 第三方 DOM 覆盖（Mantine 等）→ `src/styles/vendor-overrides.css`。
  - `src/styles/globals.css`（Tier 3 后的残余全局）：**只准删、不准加组件样式**。
- **迁一个组件到 Tailwind = 同一 commit 删它的旧 CSS**（连回规则 1）。迁完忘删 = 工作错误。
- 真要写 CSS（官方也允许的少数情况）：写进对应分工文件、**消费 token 变量**（`var(--nomi-*)`），
  不靠 `@apply`，并注释为什么 Tailwind 表达不了。

**自检**：动手往任何 `.css` 文件加规则前问一句——「这能不能用 Tailwind utility 写在组件上？」
能 → 就那么写。不能（token/全局/keyframes/vendor）→ 进对应分工文件。**别再让 CSS 文件长回去。**

## 规则 11：重要改动自己 commit + push，别攒着等人催

> 用户授权（standing instruction）：完成一个**重要/完整的改动**就**自己 commit 并 push 上去**，
> 不用每次停下来问"要不要提交"。这条**覆盖** Claude 默认的"只在用户明确要求时才 commit/push"。

**触发条件**：一个有意义的、可独立成立的改动落地且**验证通过**（见下"门槛"）。典型：
- 一个 feature chunk / bugfix / 重构 / 文档回填完成。
- 多 chunk 任务里每个 chunk（沿用既有"每 Chunk 独立 commit"的节奏）。

**铁律**：
1. **验证门槛先过再提交**：`pnpm build`（vite + electron tsc）绿 + `npx vitest run` 不回归。
   build/test 没过 → 不许 commit（更不许 push）。
2. **commit 粒度**：一个逻辑改动一个 commit，message 写清"做了什么 + 为什么 + 验证结果"，
   按既有中文 message 风格，结尾带 `Co-Authored-By`。
3. **push**：commit 后**直接 `git push`** 到当前分支（本项目工作流直接在 `main` 上推，见下"工作目录"）。
4. **只提交本次改动的文件**：用显式 `git add <files>`，别 `git add -A` 把无关的未跟踪文件
   （如别人留下的草稿 doc）也带上。

**仍然要先问、不许自动 push 的例外**（push 是 outward-facing、难撤回，这些情况停下来确认）：
- 改动**没验证通过**或你自己没把握对不对。
- 涉及**破坏性 / 不可逆**操作（删历史、force push、改 tag、发版 push tag 触发 release）。
- 用户明确说了"这个先别提交 / 我要先看看"。
- 同时改了**很多不相关的东西**揉在一起——先拆分，别一坨推上去。

**自检**：改完先问一句——「这是个完整、验证过的改动吗？」是 → 自己 commit + push，别等用户催。
不确定对不对 → 先别推，把结果摆给用户。

## 规则 12：约束代码量，防止巨型文件（棘轮只减不增）

> 反复出现且最贵的硬伤：文件越滚越大没人拦——`Scene3DFullscreen.tsx` 4598 行、
> `electron/runtime.ts` 3150 行、`styles.css` 曾滚到 8700 行死代码。巨型文件 = 关注点全
> 纠缠、改一处怕动全身、死代码没人敢删（违反规则 9）。光靠自觉拦不住，所以上自动门岗。

**铁律**：
- **单个非测试 `.ts`/`.tsx` 文件硬上限 800 行**。新文件不得超过——超了就**先拆**（按规则 9
  想清楚分层），确需保留须人工评审后才入白名单。
- **现存巨壳列入白名单并记录基线行数，棘轮只减不增**：白名单文件**只准变小、不准变大**。
  - 想往已知巨壳里加东西？先拆它 / 精简它，让净行数不增——别再喂。
  - 瘦身成功 → 把 `scripts/check-file-sizes.mjs` 里的基线下调到新值，锁定战果。
  - **目标是逐步把白名单清空**（即把所有巨壳拆掉）。
- **软目标（文化层，门岗不强制）**：日常新文件尽量 < 400~500 行；一个文件只干一件事
  （UI 归 UI、状态归 store、领域逻辑归领域层、副作用归 runtime，连回规则 9）。
- **门岗**：`pnpm run check:filesize`（`scripts/check-file-sizes.mjs`），已接入
  `.github/workflows/quality-gate.yml`。CSS 文件由规则 10 单独治理（只可减不可增），不在本门岗范围。

**自检**：写完 / 改完一个文件先问一句——「它是不是又往一个巨壳里塞？超 800 了吗？已知巨壳是不是被我喂大了？」
是 → 停下来拆，别等门岗红牌。

## 规则 13：穿透式体验走查（真·用户视角，不是 pass/fail 测试）

> 反复出现的硬伤：改完只跑单测/build 就说"验证过了"，但单测 mock 掉了真实链路，看不见
> 用户实际点进去**顺不顺、爽不爽、看着对不对**。"测试通过"≠"体验没问题"。

**定义**：用真实 app 驱动 + 多模态视觉判断，**像真实用户一样把流程点一遍**，每步截图、记录
体感、发现问题、转成优化项——这是**体验穿透**，不是功能 pass/fail。

**触发条件**：
- 声称"用户可见的改动验证过了"之前（连回规则 8——样张是动手前，走查是落地后）。
- 重构/重大改动后，确认主链路没被拆坏（功能正确 ✔ 之外，还要体验正确）。
- 定期对核心旅程（onboarding → 创作 → 画布 → 时间轴 → 导出）做真·用户体检。

**工具栈（已选定）**：**Playwright `_electron`** 驱动真实 Electron app（底层 CDP+Electron driver；
能操控主进程、截图、点击、填表、stub 原生对话框、录像/trace）。先按规则 5 查官方文档再写。
harness 在 `tests/ux/`：
- `walkthrough.mjs`——**探索式走查**（逐步截图 + dump，给多模态视觉判断；找问题用）。
- `smoke.e2e.mjs`（`pnpm run test:e2e`）——**可断言冒烟**：启动构建产物 → 断言主链路关键 UI
  真实渲染（项目库 → 开项目 → 画布工具栏/导出入口），失败即非零退出，零额度、可重复、CI-ready。
  重构/拆分后必跑它做体验级回归。（生成/导出等花额度的腿仍走探索式 + 先问用户。）

**能力边界（诚实记录，省得反复试）**：
- ① 渲染层交互 + ② 感知判断（app 体验的 ~90%）：Playwright/_electron + 多模态视觉判断**完全覆盖**，
  这是最底层、最真实的方式，不需要更"重"的技术。
- ③ 原生 OS 边界（系统文件对话框、Finder 拖拽、原生菜单/通知）：Playwright 可经
  `electronApp.evaluate` **stub `dialog.*`** 走通；要纯像素级"真用户"才需 computer-use（最 naive 但慢/脆）。
- 注：Playwright/Puppeteer/WebdriverIO 驱动渲染层底层都是 CDP——上层工具只是更工程化，不是更底层的能力。

**流程（铁律）**：
1. 起 app（Playwright `_electron`，加载 `dist-electron/main.js` 构建产物），按真实用户旅程逐步驱动。
2. **每一步截图 + 读 DOM/状态**，我以**真实用户视角**判断（不是"有没有报错"，是"顺不顺、懂不懂、舒不舒服"）。
3. 全程记进**走查日志**（`docs/qa/` 或 `docs/audit/`，带截图引用 + 每步体感 + 问题）。
4. 发现的问题按疼痛度排序，转成优化项；**用户可见的优化按规则 8（先出样张）+ 规则 7（角色评审）**走，不擅自改 UI。
5. 真实外呼/花额度的步骤（调 AI 生成/导出）默认**不擅自跑**，先问用户。

**不出走查、只跑单测就说"体验验证过了" = 工作错误。**

## 规则 14：周期性多维审计（定期回头看，别等债爆）

> 债是悄悄长的：类型门空洞、巨壳、缺测试、安全收口缺失——单看每个 commit 都没事，
> 攒一阵就成结构性问题。所以定期主动回头做一轮全量审计，而不是等出事。

**触发条件（任一满足就该跑）**：
- 距上次 `docs/audit/` 审计文档 **≥ 25 个 main commit**（`pnpm run check:audit` 会提醒，非阻断）。
- **发布新 minor 版本前**（如 0.9→0.10）。
- 任一巨壳逼近上限、lint warning / 迁移链等基线明显上涨，或直觉"乱了"。

**怎么做（一轮审计 = 这套组合拳）**：
1. **多维 subagent 深审真实代码**（规则 7 的 6 角色 + 技术栈 / 语言类型 / Agent / 架构 / 测试 / 产品 多维度），交叉比对（多维独立命中 = 最强信号）。
2. **Playwright 走查**（规则 13）补体验级回归。
3. 落 **`docs/audit/<date>-*.md`**：现状 + 分级问题（带 file:line 证据）+ 立即/中期/长期路线 + 评分。
4. **清掉清晰可改的（尤其 P0），方案级取舍留用户拍板**（规则 3）。关键论断要亲自实跑核实，不只听 agent 转述。

**自检**：开新版本 / 跑 `pnpm run check:audit` 被提醒时问一句——「该回头审一轮了吗？」是 → 别拖。
